# FASE 1 — AUTENTICACIÓN, MULTI-TENANT Y USUARIOS

## 🎯 Objetivo
Implementar autenticación segura, control de acceso por roles y aislamiento multi-empresa.

## 📚 Fuente de verdad
SRS:
- RF-01 Autenticación Segura
- RNF-03 Seguridad
- RN-06 Aislamiento por empresa
- HU-15, HU-16

## 🧩 Alcance

### Funcionalidades
- Crear empresa
- Crear usuario
- Login con JWT
- Refresh token de un solo uso
- Logout
- CRUD básico de usuarios
- Middleware de autenticación
- Middleware multi-tenant (empresa_id obligatorio)

### Estrategia 
- **JWT Strategy** 
  - Extracción de tokens desde cookies (`access_token`)
  - Extracción de tokens desde header `Authorization: Bearer <token>`
  - Validación de usuario y estado activo usando **TokenService existente**
  - Población de roles y permisos desde el payload del token

- **TokenService** 
  - Servicio compartido existente integrado
  - Generación y validación de tokens JWT
  - Generación de UUIDs

- **Guards Implementados**:
  - `JwtAuthGuard`: Protección de rutas, respeta decorador `@Public()`
  - `RolesGuard`: Validación de roles requeridos (usando enum)
  - `PermissionsGuard`: Validación de permisos específicos

- **Decoradores Útiles**:
  - `@Public()`: Marca rutas como públicas (sin autenticación)
  - `@CurrentUser()`: Inyecta el usuario autenticado
  - `@Roles(...roles)`: Requiere roles específicos (acepta enum Role)
  - `@RequirePermissions(...permissions)`: Requiere permisos específicos

### Seguridad
- Hash de contraseñas (bcrypt)
- Bloqueo tras 5 intentos fallidos
- Expiración de tokens

## 🗃️ Entidades
- empresa
- usuario
- token

## 🔌 Endpoints mínimos

-** En `auth.controller.ts`:
- ✅ POST `/api/auth/login` - Login (establece cookie httpOnly + retorna datos completos)
- ✅ POST `/api/auth/register` - Registro de estudiantes (establece cookie httpOnly + retorna datos)
- ✅ POST `/api/auth/logout` - Cierre de sesión (limpia cookie)

-** El `users.controller.ts` 
- **UUID en lugar de ID**: Todos los endpoints usan UUID para mayor seguridad
- Decoradores de roles y permisos en cada endpoint

**Ejemplo de endpoints:**
```typescript
GET    /api/users              // Lista todos (requiere users:read)
GET    /api/users/:uuid        // Obtener por UUID (requiere users:read)
POST   /api/users              // Crear (requiere ADMIN + users:create)
PUT    /api/users/:uuid        // Actualizar por UUID (requiere ADMIN + users:update)
DELETE /api/users/:uuid        // Eliminar por UUID (requiere ADMIN + users:delete)
GET    /api/users/email/:email // Buscar por email (requiere COLLABORATORS + users:read)

## ⚙️ Reglas clave
- Todo request debe incluir empresa_id (desde JWT)
- El refresh token solo puede usarse una vez
- Usuarios solo ven datos de su empresa

## ✅ Definition of Done

- Usuario puede iniciar sesión
- Tokens funcionan correctamente
- Multi-tenant aplicado en TODAS las consultas
- Roles funcionan (ADMIN, AUX, MENSAJERO)
- Sistema bloquea intentos fallidos

## 🧠 Estado final del sistema

- Sistema seguro y autenticado
- Base lista para operar con múltiples empresas
- Usuarios pueden acceder según su rol
